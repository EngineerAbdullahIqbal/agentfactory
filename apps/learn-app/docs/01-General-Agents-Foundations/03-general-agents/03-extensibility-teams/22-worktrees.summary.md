Worktrees solve the parallel editing problem -- when multiple Claude Code sessions need to make independent changes to the same codebase without overwriting each other's work. The `--worktree` (or `-w`) flag creates a complete, isolated copy of the repository at `<repo>/.claude/worktrees/<name>/` with its own branch named `worktree-<name>`, branching from the default remote branch (typically origin/main), not the current branch. If no name is provided, Claude auto-generates one. The full lifecycle is create, work, push, PR, cleanup: when you exit a session with no changes, the worktree and branch are removed automatically; when changes exist, Claude prompts you to keep or remove, where removing is destructive -- unpushed commits are permanently lost (recoverable only via git reflog within 30 days).

Subagents can also use worktree isolation by adding `isolation: worktree` to the agent's YAML frontmatter, giving each subagent its own copy of the repo that is automatically cleaned up when the subagent finishes. This complements Agent Teams (Lesson 20): teams coordinate through shared task lists and messaging, while worktrees provide filesystem isolation. You can combine both -- teams with worktrees -- when teammates need coordination AND might edit overlapping files. For manual control over the base branch or worktree placement, use `git worktree add` directly instead of the `--worktree` flag. Common failure modes include orphaned worktrees from crashed sessions (diagnosed with `git worktree list`, cleaned with `git worktree remove`), destructive cleanup of unpushed work, unexpected base branch behavior, and missing dependencies in fresh worktree directories.
