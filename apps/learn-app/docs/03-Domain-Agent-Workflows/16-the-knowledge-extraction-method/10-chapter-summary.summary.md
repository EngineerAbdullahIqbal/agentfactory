### Core Concept
The Knowledge Extraction Method is a connected chain, not a checklist: each stage produces the input the next stage requires. The methodology addresses two interdependent problems — extraction (getting tacit knowledge out of heads and documents) and validation (confirming the extracted knowledge works when encoded as SKILL.md instructions). Skipping either half produces a different kind of failure.

### Key Mental Models
- **Methodology as Chain**: Articulation gap → Method A/B extraction → SKILL.md translation → Validation scenarios → Validation loop → Shadow mode → Graduated autonomy. Each link depends on the previous one.
- **Extraction Without Validation**: Produces a SKILL.md that encodes the expert's knowledge but has unknown coverage gaps — it handles what the expert thought to mention and fails on what they did not.
- **Validation Without Extraction**: Tests instructions that do not contain the expert's actual decision-making logic — scenarios may pass, but the agent is not doing what the expert would do.
- **Domain-Agnostic Method, Domain-Specific Application**: The five questions, three passes, four validation categories, and reconciliation principle work for any domain. The specific knowledge surfaced, method priority, and reconciliation decisions differ based on professional context.

### Critical Patterns
- The credit analyst example threaded through all nine lessons was not an illustration — it was a proof of concept demonstrating that the methodology produces a testable artifact
- Every domain chapter that follows (Finance, Legal, HR, Healthcare, Architecture, Sales, Operations) applies this same methodology to a specific professional context
- The methodology is domain-agnostic but the application is domain-specific — this is why the domain chapters are necessary

### Common Mistakes
- Thinking the methodology is optional for a specific domain — every domain chapter assumes the reader can conduct an extraction and run a validation loop
- Underestimating the validation stage — the scenario set and validation loop convert a plausible draft into a production-ready file
- Treating shadow mode as a formality rather than the stage where real production inputs reveal gaps no scenario set anticipated
