# LLMs کی تین بنیادی آپریشنل رکاوٹیں

اس سے پہلے کہ آپ مؤثر طریقے سے AI ایجنٹوں کو ترتیب دے سکیں یا ڈیجیٹل FTEs بنا سکیں، آپ کو اس ٹیکنالوجی کی بنیادی نوعیت کو سمجھنے کی ضرورت ہے جس کے ساتھ آپ کام کر رہے ہیں۔ بڑے لسانی ماڈلز (LLMs) — جو کلاڈ اوپس 4.5، GPT-5.2، جیمنی 3، اور ان پر بنے ہر AI کوڈنگ ایجنٹ کو طاقت دینے والے استدلال کے انجن ہیں — تین بنیادی رکاوٹوں کے تحت کام کرتے ہیں جو آپ ان کے ساتھ کام کرنے کے طریقے کے بارے میں ہر چیز کو تشکیل دیتے ہیں۔ ان رکاوٹوں کو غلط سمجھنا زیادہ تر مایوسیوں کی بنیادی وجہ ہے جو ڈویلپرز کو AI ٹولز کے ساتھ ہوتی ہے۔

یہ ٹھیک کرنے کے لیے کیڑے یا کام کرنے کے لیے حدود نہیں ہیں۔ یہ LLMs کے کام کرنے کے طریقے کی بنیادی خصوصیات ہیں۔ اس کتاب میں ہر طریقہ کار — اسپیک سے چلنے والی ترقی سے لے کر سیاق و سباق کی انجینئرنگ سے لے کر OODA لوپ تک — ان رکاوٹوں کی وجہ سے موجود ہے۔ انہیں سمجھنا آپ کو ٹیکنالوجی سے لڑنے والے سے اس کے ساتھ کام کرنے والے میں بدل دیتا ہے۔

## رکاوٹ 1: LLMs بے ریاست ہیں۔

**حقیقت**: جب بھی آپ کسی LLM کو پیغام بھیجتے ہیں، اس کے پاس پچھلی بات چیت کی کوئی یادداشت نہیں ہوتی ہے۔ ماڈل آپ کی آخری گفتگو، آپ کی ترجیحات، یا یہاں تک کہ آپ نے پانچ منٹ پہلے اسی چیٹ سیشن میں کیا کہا تھا اسے "یاد" نہیں رکھتا ہے۔ یہ ہر ماڈل کے لیے سچ ہے — کلاڈ اوپس 4.5، GPT-5.2، جیمنی 3 — چاہے وہ کتنے ہی جدید کیوں نہ ہوں۔

**"بے ریاست" کا اصل مطلب کیا ہے**:

ایل ایل ایم کو ہر API کال کو ایک مکمل نئی شروعات کے طور پر سوچیں—جیسے کسی ایسے شخص سے بات کرنا جسے مکمل بھولنے کی بیماری ہو۔ ماڈل آپ کا پیغام وصول کرتا ہے، اس پر کارروائی کرتا ہے، جواب دیتا ہے، اور پھر فوراً سب کچھ بھول جاتا ہے۔ اگلا پیغام ایک خالی سلیٹ پر آتا ہے۔

**یادداشت کا وہم**:

جب آپ ChatGPT یا Claude میں گفتگو کرتے ہیں، تو ایسا *لگتا* ہے کہ ماڈل پہلے کے پیغامات کو یاد رکھتا ہے۔ لیکن یہاں اصل میں کیا ہو رہا ہے: ایپلی کیشن (ماڈل نہیں) آپ کی گفتگو کی تاریخ کو ذخیرہ کرتی ہے، اور ہر نئے پیغام کے ساتھ، پوری گفتگو ماڈل کو دوبارہ بھیجی جاتی ہے۔ ماڈل ہر بار شروع سے پوری گفتگو پڑھتا ہے۔

```
Message 1: You -> "Hi, my name is Maya"
[Model processes, responds, forgets everything]

Message 2: You -> "What's my name?"
[Application sends: "Hi, my name is Maya" + "What's my name?"]
[Model reads full history, responds "Maya", forgets everything]

Message 3: You -> "I'm building a registration system"
[Application sends: Message 1 + Message 2 + Response 2 + Message 3]
[Model reads ENTIRE history from scratch, responds, forgets everything]
```

**یہ AI-آبائی ترقی کے لیے کیوں اہمیت رکھتا ہے**:

1. **سیاق و سباق کو واضح طور پر فراہم کیا جانا چاہئے**: ماڈل آپ کے پروجیکٹ کے بارے میں نہیں جانتا جب تک کہ آپ اسے نہ بتائیں—ہر بار۔ یہی وجہ ہے کہ AGENTS.md موجود ہے: یہ مستقل سیاق و سباق فراہم کرتا ہے جو ہر تعامل میں بھرا جاتا ہے۔

2. **تفصیلات اختیاری نہیں ہیں—وہ ضروری ہیں**: جب آپ تفصیلی تفصیلات فراہم کرتے ہیں، تو آپ صرف مکمل نہیں ہو رہے ہوتے ہیں؛ آپ ماڈل کو وہ واحد معلومات دے رہے ہوتے ہیں جو اسے آپ کی ضروریات کے بارے میں کبھی حاصل ہوگی۔

3. **سیشن کا تسلسل ایک ایپلی کیشن کی خصوصیت ہے، ماڈل کی خصوصیت نہیں**: کلاڈ کوڈ جیسے اوزار ہر تعامل کے ساتھ اسے دوبارہ داخل کرکے سیاق و سباق کو برقرار رکھتے ہیں۔ اسے سمجھنا آپ کو نظام کے اندر کام کرنے میں مدد کرتا ہے بجائے اس کے کہ ایسی "یادداشت" کی توقع کریں جو موجود نہیں ہے۔

4. **طویل منصوبوں کے لیے واضح ریاستی انتظام کی ضرورت ہوتی ہے**: پیچیدہ منصوبوں کے لیے، آپ کو فائلوں (جیسے SPEC.md یا PROJECT_CONTEXT.md) میں تفصیلات، فیصلے، اور سیاق و سباق کو برقرار رکھنے کی ضرورت ہے جنہیں ہر سیشن میں دوبارہ بھرا جاسکتا ہے۔

**عملی مضمرات**:

| آپ کیا توقع کرسکتے ہیں | اصل میں کیا ہوتا ہے | کیسے ڈھالیں |
|----------------------|----------------------|--------------|
| "یہ میرا کوڈنگ اسٹائل یاد رکھتا ہے" | ماڈل کے پاس پچھلے سیشنز کی کوئی یادداشت نہیں ہے۔ | AGENTS.md یا تفصیلات میں اسٹائل گائیڈز فراہم کریں۔ |
| "یہ میرا پروجیکٹ جانتا ہے" | ہر پیغام تازہ شروع ہوتا ہے۔ | ماڈل کو اپنے کوڈ بیس تک رسائی دیں؛ AI-فرسٹ IDEs اور/یا کوڈنگ ایجنٹس استعمال کریں۔ |
| "اس نے ہماری آخری گفتگو سے سیکھا" | سیشنز کے درمیان کوئی سیکھنے کا عمل نہیں ہوتا ہے۔ | فیصلوں کی دستاویز کریں اور سیاق و سباق کو دوبارہ داخل کریں۔ |
| "یہ اس ترجیح کو یاد رکھے گا" | ترجیحات فوراً بھول جاتی ہیں۔ | مستقل کنفیگریشن فائلوں میں ترجیحات کو انکوڈ کریں۔ |

## رکاوٹ 2: LLMs امکانی ہیں، قطعی نہیں

**حقیقت**: ایک ہی ان پٹ دیے جانے پر، ایک LLM اکثر مختلف آؤٹ پٹ پیدا کرے گا۔ روایتی سافٹ ویئر کے برعکس جو مستقل نتائج لوٹاتا ہے، LLMs امکانی تقسیم سے نمونے لے کر جوابات تیار کرتے ہیں، جس سے موروثی تغیر پیدا ہوتا ہے۔ یہ کلاڈ اوپس 4.5، GPT-5.2، اور جیمنی 3 پر یکساں طور پر لاگو ہوتا ہے — امکانی نوعیت تمام ٹرانسفارمر پر مبنی ماڈلز کے کام کرنے کے طریقے کے لیے بنیادی ہے، نہ کہ پرانی نسلوں کی کوئی حد۔

**"امکانی" کا اصل مطلب کیا ہے**:

جب ایک LLM متن تیار کرتا ہے، تو اس کے پاس کہیں ایک "درست" جواب ذخیرہ نہیں ہوتا ہے۔ اس کے بجائے، ہر قدم پر، یہ بہت سے ممکنہ اگلے ٹوکنز (الفاظ یا الفاظ کے حصے) کے امکان کا حساب لگاتا ہے اور ایک کو منتخب کرتا ہے۔ اس انتخاب کے عمل میں بے ترتیب پن شامل ہوتا ہے۔

```
Prompt: "The best programming language for web development is"

Run 1: "JavaScript, because..." (probability sampled)
Run 2: "TypeScript, given..." (different sample from same distribution)
Run 3: "Python with Django..." (yet another valid sample)
```

ہر جواب درست اور معقول ہے—لیکن وہ مختلف ہیں۔ ماڈل غلطیاں نہیں کر رہا ہے؛ یہ بالکل ویسا ہی کام کر رہا ہے جیسا کہ ڈیزائن کیا گیا ہے۔

**درجہ حرارت کا عنصر**:

تمام فرنٹیئر ماڈلز—کلاڈ اوپس 4.5، GPT-5.2، اور جیمنی 3—میں ایک "درجہ حرارت" کا پیرامیٹر ہوتا ہے جو یہ کنٹرول کرتا ہے کہ بے ترتیب پن آؤٹ پٹ کو کتنا متاثر کرتا ہے:

- **درجہ حرارت = 0**: ماڈل ہمیشہ سب سے زیادہ امکان والے ٹوکن کا انتخاب کرتا ہے۔ زیادہ قطعی، لیکن دہرایا جانے والا اور کم تخلیقی ہوسکتا ہے۔
- **درجہ حرارت = 0.7** (عام ڈیفالٹ): متوازن تخلیقی صلاحیت اور ہم آہنگی۔ ایک ہی پرامپٹ مختلف لیکن سمجھدار جوابات دے گا۔
- **درجہ حرارت = 1.0+**: اعلی تخلیقی صلاحیت، لیکن غیر مربوط یا غیر متوقع آؤٹ پٹ کا بڑھتا ہوا خطرہ۔

یہاں تک کہ درجہ حرارت = 0 پر بھی، فلوٹنگ پوائنٹ کے حسابات اور بیچنگ کی وجہ سے لطیف تغیرات ہوسکتے ہیں۔

**یہ AI-آبائی ترقی کے لیے کیوں اہمیت رکھتا ہے**:

1. **آپ ایک جیسے پرامپٹس سے ایک جیسے کوڈ کی توقع نہیں کرسکتے ہیں**: ایک ہی تفصیلات کو AI ایجنٹ کے ذریعے دو بار چلانے سے دو مختلف (لیکن دونوں درست) نفاذ پیدا ہوسکتے ہیں۔ یہ کوئی بگ نہیں ہے—یہ ٹیکنالوجی کی نوعیت ہے۔

2. **توثیق ضروری ہوجاتی ہے، اختیاری نہیں**: چونکہ آؤٹ پٹ مختلف ہوتے ہیں، اس لیے آپ کو اس بات کی توثیق کرنی ہوگی کہ کوئی بھی دیا گیا آؤٹ پٹ آپ کی تفصیلات پر پورا اترتا ہے۔ یہی وجہ ہے کہ SDD کا "توثیق" مرحلہ بیوروکریسی نہیں ہے—یہ ایک بنیادی ضرورت ہے۔

3. **تفصیلات اینکرنگ فراہم کرتی ہیں**: جبکہ آؤٹ پٹ مختلف ہوتے ہیں، ایک واضح تفصیلات درست آؤٹ پٹ کی *جگہ* کو محدود کرتی ہے۔ مبہم پرامپٹس بے حد مختلف نتائج دیتے ہیں؛ عین مطابق تفصیلات قابل قبول حدود کے اندر تغیرات پیدا کرتی ہیں۔

4. **ٹیسٹ سے چلنے والی ترقی نئی اہمیت اختیار کرتی ہے**: ٹیسٹ *غیر متغیر* کی وضاحت کرتے ہیں جو نفاذ کی تبدیلی سے قطع نظر برقرار رہنا چاہئے۔ جب AI کوڈ تیار کرتا ہے، تو ٹیسٹ اس بات کی تصدیق کرتے ہیں کہ امکانی آؤٹ پٹ قطعی ضروریات کو پورا کرتا ہے۔

5. **تکرار معمول ہے، استثناء نہیں**: کامل تفصیلات کے ساتھ بھی 1-2 اصلاحی چکروں کی منصوبہ بندی کریں۔ پہلا آؤٹ پٹ 95% درست ہوسکتا ہے لیکن اسے ایڈجسٹمنٹ کی ضرورت پڑسکتی ہے۔ یہ معمول ہے—اسے اپنے ورک فلو میں شامل کریں۔

**عملی مضمرات**:

| آپ کیا توقع کرسکتے ہیں | اصل میں کیا ہوتا ہے | کیسے ڈھالیں |
|----------------------|----------------------|--------------|
| "ایک ہی پرامپٹ، ایک ہی کوڈ" | ہر رن میں مختلف آؤٹ پٹ | تغیر کو محدود کرنے کے لیے تفصیلات استعمال کریں؛ آؤٹ پٹ کی توثیق کریں۔ |
| "اس نے مجھے یہ آخری بار دیا تھا" | مستقل مزاجی کی کوئی ضمانت نہیں۔ | ورژن کنٹرول جو کام کرتا ہے؛ دوبارہ پیدا کرنے کی صلاحیت پر بھروسہ نہ کریں۔ |
| "اس بگ کو ٹھیک کریں" وہی ٹھیک دیتا ہے | متعدد درست اصلاحات موجود ہیں۔ | ہر ٹھیک کا جائزہ لیں؛ ضروریات کے خلاف ٹیسٹ کریں۔ |
| "اس ٹیسٹ سویٹ کو دو بار تیار کریں" | مختلف ٹیسٹ کیسز سامنے آتے ہیں۔ | مطلوبہ کوریج کو واضح طور پر بیان کریں؛ متعدد رن کے بہترین کو ضم کریں۔ |

**امکانی آؤٹ پٹ کی طاقت**:

یہ خالصتاً ایک حد نہیں ہے—یہ ایک طاقت بھی ہے۔ امکانی نسل کا مطلب ہے:

- **تخلیقی مسئلہ حل کرنا**: ماڈل ایسے طریقوں کی تجویز کرسکتا ہے جن پر آپ نے غور نہیں کیا تھا۔
- **متعدد درست حل**: آپ کئی نفاذات تیار کرسکتے ہیں اور بہترین کا انتخاب کرسکتے ہیں۔
- **حل کی جگہ کی تلاش**: ایک پرامپٹ کو متعدد بار چلانے سے کسی مسئلے پر مختلف زاویے ظاہر ہوسکتے ہیں۔

کلید اس خصوصیت کے ساتھ کام کرنا ہے بجائے اس کے کہ اس سے لڑیں۔

## رکاوٹ 3: سیاق و سباق محدود ہے، لامحدود نہیں

**حقیقت**: LLMs کے پاس ایک مقررہ "سیاق و سباق کی کھڑکی" ہوتی ہے—متن کی زیادہ سے زیادہ مقدار جسے وہ ایک ساتھ پروسیس کرسکتے ہیں۔ یہ کھڑکی ہر چیز کو ذخیرہ کرتی ہے: سسٹم پرامپٹ، آپ کی گفتگو کی تاریخ، اپ لوڈ کردہ فائلیں، اور ماڈل کا جواب۔ ایک بار بھر جانے پر، پرانا مواد باہر دھکیل دیا جاتا ہے یا چھوٹا کردیا جاتا ہے۔

**"سیاق و سباق کی کھڑکی" کا اصل مطلب کیا ہے**:

سیاق و سباق کی کھڑکی کو ماڈل کی ورکنگ میموری کے طور پر سوچیں—ہر وہ چیز جسے وہ کسی بھی لمحے "دیکھ" سکتا ہے۔ 2026 کے اوائل تک، فرنٹیئر ماڈلز میں متاثر کن لیکن پھر بھی محدود سیاق و سباق کی کھڑکیاں ہیں:

| ماڈل | سیاق و سباق کی کھڑکی | تقریباً مساوی |
|-------|---------------|----------------------|
| GPT-5.2 (OpenAI) | 256K ٹوکنز | ~200,000 الفاظ / ~600 صفحات |
| کلاڈ اوپس 4.5 (اینتھروپک) | 200K ٹوکنز | ~150,000 الفاظ / ~500 صفحات |
| جیمنی 3 پرو (گوگل) | 2M ٹوکنز | ~1,500,000 الفاظ / ~5,000 صفحات |

یہاں تک کہ جیمنی 3 کی متاثر کن 2 ملین ٹوکن والی کھڑکی—فرنٹیئر ماڈلز میں سب سے بڑی—انٹرپرائز کوڈ بیسز پر تیزی سے بھر جاتی ہے۔ اور بڑی سیاق و سباق کی کھڑکیوں کے ساتھ تجارت ہوتی ہے: بڑھتی ہوئی تاخیر، زیادہ لاگت، اور ممکنہ "درمیان میں کھو جانے" کے اثرات جہاں بہت لمبے سیاق و سباق کے مرکز میں موجود معلومات کو شروع یا آخر میں موجود مواد سے کم توجہ ملتی ہے۔

یہ تعداد بہت بڑی لگتی ہے—جب تک کہ آپ کو یہ احساس نہ ہو کہ انہیں کیا بھرتا ہے:

```
System prompt (AGENTS.md, skills, instructions): 5,000-20,000 tokens
Your specification: 1,000-5,000 tokens
Conversation history: 2,000-50,000 tokens
Relevant code files: 10,000-100,000 tokens
Model's response: 1,000-10,000 tokens
-------------------------------------------
TOTAL: Can exceed limit quickly on complex projects
```

**محدود سیاق و سباق کے نتائج**:

1. **معلومات ضائع ہوجاتی ہیں**: لمبی گفتگو میں، ابتدائی پیغامات کو چھوٹا یا خلاصہ کیا جاسکتا ہے۔ ماڈل "بھول جاتا ہے" اس لیے نہیں کہ اس کی یادداشت خراب ہے، بلکہ اس لیے کہ معلومات اب فٹ نہیں ہوتی ہیں۔

2. **بڑے کوڈ بیس فٹ نہیں ہوتے**: ایک معتدل پیچیدہ پروجیکٹ میں 50,000-500,000+ لائنوں کا کوڈ ہوسکتا ہے۔ یہاں تک کہ سب سے بڑی سیاق و سباق کی کھڑکیاں بھی یہ سب نہیں رکھ سکتیں۔

3. **سیاق و سباق صفر کا مجموعہ ہے**: گفتگو کی تاریخ پر خرچ کیا گیا ہر ٹوکن کوڈ، تفصیلات، یا جواب کی نسل کے لیے دستیاب نہیں ہے۔

**یہ AI-آبائی ترقی کے لیے کیوں اہمیت رکھتا ہے**:

1. **سیاق و سباق کی انجینئرنگ ایک بنیادی مہارت ہے**: یہ فیصلہ کرنا کہ سیاق و سباق کی کھڑکی میں کیا جاتا ہے—اور کیا نہیں—براہ راست آؤٹ پٹ کے معیار کو متاثر کرتا ہے۔ یہی وجہ ہے کہ MCP (ماڈل سیاق و سباق پروٹوکول) اور ترقی پسند مہارت کی لوڈنگ موجود ہے۔

2. **AI-فرسٹ IDEs سیاق و سباق کے مسائل حل کرتے ہیں**: کرسر اور ونڈ سرف جیسے اوزار صرف آپ کا کوڈ ماڈل کو نہیں بھیجتے ہیں۔ وہ ذہانت سے *متعلقہ* کوڈ کا انتخاب کرتے ہیں—وہ فائلیں جن میں آپ ترمیم کر رہے ہیں، درآمدات، متعلقہ افعال—محدود سیاق و سباق کی قدر کو زیادہ سے زیادہ کرنے کے لیے۔

3. **تفصیلات ارادے کو سکیڑتی ہیں**: ایک اچھی طرح سے لکھی گئی تفصیلات بکھری ہوئی گفتگو سے کم ٹوکنز میں ضروریات کو پہنچاتی ہے۔ "ان 10 معیارات کے ساتھ ایک رجسٹریشن سسٹم بنائیں" 50-پیغام کے آگے پیچھے سے زیادہ موثر ہے جس نے ان معیارات کو دریافت کیا۔

4. **پروجیکٹ کی ساخت اہمیت رکھتی ہے**: آپ کوڈ کو کیسے منظم کرتے ہیں اس پر اثر پڑتا ہے کہ AI کس سیاق و سباق تک رسائی حاصل کرسکتا ہے۔ چھوٹی، اچھی طرح سے نامزد فائلیں جن کی واضح ذمہ داریاں ہوں، یک سنگی فائلوں کے مقابلے میں منتخب طور پر شامل کرنا آسان ہے۔

5. **گفتگو کا انتظام حکمت عملی بن جاتا ہے**: طویل ڈیبگنگ سیشنز کو غیر متعلقہ تاریخ کے ہزاروں ٹوکنز کو گھسیٹنے کے بجائے صرف متعلقہ سیاق و سباق کے ساتھ ایک تازہ گفتگو شروع کرکے "ری سیٹ" کرنے کی ضرورت پڑسکتی ہے۔

**عملی مضمرات**:

| آپ کیا توقع کرسکتے ہیں | اصل میں کیا ہوتا ہے | کیسے ڈھالیں |
|----------------------|----------------------|--------------|
| "یہ میرا پورا کوڈ بیس دیکھ سکتا ہے" | صرف منتخب فائلیں سیاق و سباق میں فٹ ہوتی ہیں۔ | سمارٹ سیاق و سباق کے انتخاب کے ساتھ AI-فرسٹ IDEs استعمال کریں۔ |
| "یہ ہماری پوری گفتگو کو یاد رکھتا ہے" | پرانے پیغامات چھوٹے ہوجاتے ہیں۔ | گفتگو کو مرکوز رکھیں؛ نئے موضوعات کے لیے تازہ شروع کریں۔ |
| "زیادہ سیاق و سباق ہمیشہ بہتر ہوتا ہے" | شور متعلقہ معلومات کو کم کرتا ہے۔ | سیاق و سباق کو احتیاط سے درست کریں؛ مقدار پر معیار۔ |
| "یہ میری ریپو میں سب کچھ جانتا ہے" | سیاق و سباق کی کھڑکی کی سخت حدود ہیں۔ | منتخب شمولیت کے لیے ریپوز کی ساخت؛ واقفیت کے لیے AGENTS.md استعمال کریں۔ |

**سیاق و سباق کے انتظام کی حکمت عملی**:

**تفصیلات کے لیے**: سب سے اہم رکاوٹوں کو سامنے لوڈ کریں۔ اگر سیاق و سباق کو چھوٹا کیا جاتا ہے، تو اوپر کی اہم ضروریات بچ جاتی ہیں جبکہ نیچے کی اچھی چیزیں ضائع ہوسکتی ہیں۔

**کوڈ کے لیے**: جب AI کو فائل سسٹم تک رسائی حاصل ہو تو پورے مواد کو پیسٹ کرنے کے بجائے راستے سے فائلوں کا حوالہ دیں۔ اسے وہ چیز بازیافت کرنے دیں جس کی اسے ضرورت ہے۔

**گفتگو کے لیے**: جب گفتگو بے قابو ہوجائے، تو پیشرفت کا خلاصہ کریں اور تازہ شروع کریں:

```
"Let's reset. We're building a user registration system.
We've decided on:
- PostgreSQL database
- bcrypt for passwords
- AWS SES for email

Now I need help with the rate limiting implementation."
```

**پروجیکٹس کے لیے**: ایک PROJECT_CONTEXT.md فائل کو برقرار رکھیں جو اہم فیصلوں، فن تعمیر کے جائزہ، اور موجودہ حیثیت کو حاصل کرتی ہے۔ یہ نئے سیشنز کے لیے آپ کی "ریاستی انجیکشن" بن جاتی ہے۔

## تین رکاوٹیں کیسے آپس میں جڑی ہوئی ہیں۔

یہ رکاوٹیں الگ تھلگ نہیں ہیں—یہ مرکب اور تعامل کرتی ہیں:

**بے ریاست + محدود سیاق و سباق**: چونکہ ماڈل پچھلے سیشنز کو یاد نہیں رکھتا، اس لیے آپ کو ہر بار سیاق و سباق کو دوبارہ داخل کرنا ہوگا۔ لیکن سیاق و سباق محدود ہے، لہذا آپ کو *موثر طریقے سے* دوبارہ داخل کرنا ہوگا۔ یہی وجہ ہے کہ AGENTS.md فائلیں جامع ہونے کے بجائے جامع ہیں۔

**امکانی + بے ریاست**: ہر سیشن تازہ شروع ہوتا ہے اور متغیر آؤٹ پٹ پیدا کرتا ہے۔ مستقل حالت کے بغیر، آپ اس بات کی ضمانت بھی نہیں دے سکتے کہ ماڈل کسی مسئلے کو دو بار ایک ہی طریقے سے حل کرے گا۔ یہی وجہ ہے کہ ورژن کنٹرول اور فیصلوں کی واضح دستاویزات اہمیت رکھتی ہیں۔

**امکانی + محدود سیاق و سباق**: جب سیاق و سباق محدود ہوتا ہے، تو ماڈل کے پاس اپنی امکانی نسل کو اینکر کرنے کے لیے کم معلومات ہوتی ہیں۔ مبہم تفصیلات کے علاوہ محدود سیاق و سباق بے حد مختلف آؤٹ پٹ دیتا ہے۔ سیاق و سباق کی رکاوٹوں کے اندر واضح تفصیلات حدود کے اندر مفید تغیر پیدا کرتی ہیں۔

## طریقہ کار کا جواب

اس کتاب میں ہر مشق ان تین رکاوٹوں کی وجہ سے موجود ہے:

| رکاوٹ | کے لیے ضرورت پیدا کرتا ہے۔ | کی طرف سے خطاب کیا |
|-----------|---------------------|--------------|
| **بے ریاست** | مستقل سیاق و سباق جو سیشنز سے بچ جاتا ہے۔ | AGENTS.md، SPEC.md، MCP، ہنر |
| **امکانی** | متغیر آؤٹ پٹ کی توثیق | SDD کا توثیق مرحلہ، TDD، کوالٹی گیٹس |
| **محدود سیاق و سباق** | ضروریات کی موثر نمائندگی | تفصیلات-پہلی سوچ، سیاق و سباق کی انجینئرنگ |

**اسپیک سے چلنے والی ترقی** تینوں کو حل کرتی ہے:
- تفصیلات سیشنز میں برقرار رہتی ہیں (بے ریاستی کا مقابلہ کرتی ہیں)
- واضح تفصیلات امکانی آؤٹ پٹ کو محدود کرتی ہیں (تغیر کو کم کرتی ہیں)
- مختصر تفصیلات سیاق و سباق کی کھڑکی کے استعمال کو زیادہ سے زیادہ کرتی ہیں (حدود کا احترام کرتی ہیں)

**AGENTS.md** بے ریاستی کا براہ راست جواب ہے—ایک مستقل فائل جسے ہر سیشن کو مستقل بنیادی علم دینے کے لیے سیاق و سباق میں داخل کیا جاتا ہے۔

**MCP (ماڈل سیاق و سباق پروٹوکول)** ایجنٹوں کو ہر چیز کو پہلے سے درکار کرنے کے بجائے متحرک طور پر معلومات بازیافت کرنے کی اجازت دے کر سیاق و سباق کی حد کو حل کرتا ہے۔

**ٹیسٹ سے چلنے والی ترقی** امکانی آؤٹ پٹ کو غیر متغیر (ٹیسٹ) کی وضاحت کرکے قبول کرتی ہے جو کسی بھی درست نفاذ کو پورا کرنا چاہئے، چاہے وہ کیسے ہی تیار کیا گیا ہو۔

## ڈویلپر کا ذہنی ماڈل

ان رکاوٹوں کو سمجھنا AI تعاون کے بارے میں آپ کی سوچ کو بدل دیتا ہے:

**پرانا ذہنی ماڈل (غلط)**:
- "AI ایک باخبر ساتھی کی طرح ہے جو ہمارے ساتھ کام کو یاد رکھتا ہے"
- "اگر میں ایک بار کچھ سمجھاؤں تو وہ جانتا ہے"
- "ایک ہی پرامپٹ مجھے ایک ہی نتیجہ دینا چاہئے"

**نیا ذہنی ماڈل (درست)**:
- "AI ایک شاندار ماہر کی طرح ہے جسے بھولنے کی بیماری ہے جسے میں ہر بار شروع سے بریف کرتا ہوں"
- "مجھے ہر سیشن میں تمام متعلقہ سیاق و سباق کو مختصر طور پر فراہم کرنا ہوگا"
- "میں جو چاہتا ہوں اس کی وضاحت کرتا ہوں، جو مجھے ملتا ہے اس کی توثیق کرتا ہوں، اور تکرار کی توقع کرتا ہوں"

یہ ذہنی ماڈل مایوس کن نہیں ہے—یہ عملی ہے۔ جب آپ رکاوٹوں کو سمجھتے ہیں، تو آپ ان سے لڑنا چھوڑ دیتے ہیں اور ایسے ورک فلو ڈیزائن کرنا شروع کردیتے ہیں جو ان کے ساتھ کام کرتے ہیں۔ تب ہی AI مایوس کن ہونے کے بجائے حقیقی طور پر نتیجہ خیز ہوجاتا ہے۔

---

### ہیلوسینیشن کا خطرہ

LLMs اعتماد کے ساتھ ایسا کوڈ تیار کرسکتے ہیں جو درست لگتا ہے لیکن اس میں لطیف کیڑے ہوتے ہیں، غیر موجود APIs کا حوالہ دیتے ہیں، یا ایسی منطق کو نافذ کرتے ہیں جو آپ کے ارادے سے میل نہیں کھاتی ہے۔ 
یہ جھوٹ نہیں ہے—یہ امکانی نوعیت ہے جو شماریاتی نمونوں سے پراعتماد لگنے والے آؤٹ پٹ پیدا کرتی ہے۔ یہی وجہ ہے کہ توثیق اختیاری نہیں ہے: آپ تصدیق کے بغیر AI سے تیار کردہ کوڈ پر بھروسہ نہیں کرسکتے ہیں۔

---

### سیاق و سباق اور لاگت

سیاق و سباق کی کھڑکی میں ہر ٹوکن کی قیمت ہوتی ہے۔ فرنٹیئر ماڈل APIs فی ان پٹ اور آؤٹ پٹ ٹوکن چارج کرتی ہیں۔ ایک ناقص منظم سیاق و سباق (غیر متعلقہ فائلوں کو بھرنا، طویل گفتگو کی تاریخیں) براہ راست لاگت میں اضافہ کرتا ہے۔ موثر تفصیلات اور سمارٹ سیاق و سباق کی انجینئرنگ صرف معیار کے بارے میں نہیں ہیں—وہ بڑے پیمانے پر معاشیات کے بارے میں ہیں۔

## AI کے ساتھ کوشش کریں۔

**رکاوٹ کی تلاش کی مشق**

ان رکاوٹوں کو خود دریافت کرنے کے لیے اپنے AI ساتھی کا استعمال کریں:

```
I want to understand how LLMs work. Let's do an experiment.

1. STATELESSNESS: Ask me 3 questions about myself (name, favorite color, current project).
   I'll answer. Then I want you to explain: Do you actually "remember" my answers,
   or are you reading them from the conversation history? What would happen if
   this conversation's history was cleared?

2. PROBABILISTIC: Generate a Python function to check if a number is prime.
   Then generate it again without looking at the first version.
   Compare the two implementations—why are they different?

3. CONTEXT LIMITS: Explain what happens if I paste an entire 50,000-line codebase
   into this conversation. What would get lost? How should I share large codebases
   with you instead?

Help me build intuition for how these constraints affect how I should work with you.
```

**آپ کیا سیکھ رہے ہیں**: رکاوٹوں کے ساتھ براہ راست تجربہ۔ یہ گہری سمجھ پیدا کرتا ہے جو ان کے بارے میں پڑھنا فراہم نہیں کرسکتا۔

**سیاق و سباق کی انجینئرنگ کی مشق**

```
I'm going to practice context engineering. Here's a challenge:

I have a user authentication system with:
- 15 source files (about 3,000 lines total)
- 5 database migrations
- 3 configuration files
- An existing README

I want your help debugging a session management bug.

Help me figure out:
1. What's the minimum context you need to help effectively?
2. What information would just add noise?
3. How should I structure my question to maximize your helpfulness
   within context limits?
4. If you could only see 3 files, which would be most valuable?

Don't make assumptions about my code—ask clarifying questions
that help you understand what context you actually need.
```

**آپ کیا سیکھ رہے ہیں**: سیاق و سباق کی درستگی کے بارے میں کیسے سوچیں۔ یہ مہارت براہ راست اس بات پر اثر انداز ہوتی ہے کہ آپ کا AI تعاون کتنا موثر ہوجاتا ہے۔
